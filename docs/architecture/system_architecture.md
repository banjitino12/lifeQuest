# LifeQuest 系统架构设计

创建日期：2026-06-08

## 1. 架构目标

LifeQuest 的系统架构围绕 MVP 闭环设计：

```text
记录 -> 解析 -> 评分 -> 属性变化 -> 游戏化反馈 -> 明日任务 -> 趋势复盘
```

架构目标：

- 支持用户注册登录、目标设置、每日记录、结算反馈和趋势展示。
- 保证规则评分稳定可解释，不让 LLM 直接决定最终分数。
- 将 LLM 能力封装在后端统一服务中，便于替换模型、增加降级策略和审计调用结果。
- 支持 MySQL、MongoDB、Redis 的职责分离。
- 保持 MVP 结构清晰，避免过早引入复杂社交、装备或多 Agent 系统。

## 2. 总体架构

项目采用前后端分离架构。

```text
┌────────────────────────────────────────────────────────────┐
│                         Client                             │
│  Vue 3 + TypeScript + UI Components + ECharts              │
└───────────────────────────┬────────────────────────────────┘
                            │ HTTP / JSON
┌───────────────────────────▼────────────────────────────────┐
│                      Backend API                           │
│  Spring Boot + Spring Security + JWT                       │
├────────────────────────────────────────────────────────────┤
│ Controller Layer                                            │
│  Auth / User / Profile / DailyLog / Settlement / Trend      │
├────────────────────────────────────────────────────────────┤
│ Application Service Layer                                   │
│  Daily Record Orchestration / Settlement Orchestration      │
├────────────────────────────────────────────────────────────┤
│ Domain Service Layer                                        │
│  Scoring / Attribute / Gamification / Route / LLM           │
├────────────────────────────────────────────────────────────┤
│ Repository & Integration Layer                              │
│  MySQL Repository / Mongo Repository / Redis / LLM Client   │
└───────────────┬────────────────┬────────────────┬──────────┘
                │                │                │
        ┌───────▼───────┐ ┌──────▼──────┐ ┌──────▼──────┐
        │    MySQL      │ │  MongoDB    │ │    Redis    │
        │ structured    │ │ long text   │ │ cache/rate  │
        │ business data │ │ llm output  │ │ session     │
        └───────────────┘ └─────────────┘ └─────────────┘
```

## 3. 前端架构

### 3.1 技术选型

- Vue 3：前端应用框架
- TypeScript：类型约束
- Element Plus / Naive UI：基础 UI 组件
- ECharts：趋势图、雷达图等数据可视化
- Pinia：状态管理，后续初始化时优先采用
- Vue Router：页面路由

### 3.2 前端目录职责

```text
frontend/src/
├── api/             # 后端接口请求封装
├── assets/          # 静态资源
├── components/      # 通用组件
│   ├── charts/      # 图表组件
│   ├── game/        # 游戏化展示组件
│   └── layout/      # 布局组件
├── router/          # 路由定义
├── stores/          # 用户、记录、结算等状态
├── styles/          # 全局样式
├── types/           # API 和业务类型定义
├── utils/           # 工具函数
└── views/           # 页面级组件
```

### 3.3 前端页面分层

- `views/auth`：登录、注册、找回密码。
- `views/dashboard`：成长首页，展示今日评分、等级、路线、连续记录天数和属性雷达图。
- `views/daily-log`：每日记录，支持结构化表单和自然语言日志输入。
- `views/settlement`：每日结算，展示评级、经验、属性变化、事件、反馈和明日任务。
- `views/route`：成长路线，展示章节进度、当前关卡和下一目标。
- `views/trends`：成长趋势，展示评分、学习、睡眠、情绪和属性变化趋势。
- `views/weekly`：周报，展示本周总结、主要问题、成长点和下周建议。

### 3.4 前端数据原则

- 前端只展示评分结果和解释，不作为最终评分来源。
- 前端可以做表单校验，但核心业务校验必须在后端重复执行。
- 用户原始日志仅在用户个人页面内展示，不进入公开页面或共享组件。
- 趋势图只消费后端聚合后的数据，避免在前端复制复杂统计逻辑。

## 4. 后端架构

### 4.1 技术选型

- Spring Boot：后端应用框架
- Spring Security + JWT：登录鉴权和接口保护
- MyBatis Plus 或 Spring Data JPA：结构化数据访问
- MySQL：核心业务数据
- MongoDB：长文本日志、LLM 原始反馈、周报内容
- Redis：近期状态缓存、限流、会话或 Token 黑名单

### 4.2 后端分层

```text
Controller
  -> Application Service
    -> Domain Service
      -> Repository / External Client
```

#### Controller Layer

职责：

- 接收 HTTP 请求
- 参数校验
- 获取当前登录用户
- 调用应用服务
- 返回统一响应结构

#### Application Service Layer

职责：

- 编排完整业务流程
- 管理事务边界
- 聚合多个领域服务结果
- 处理异步任务触发

典型服务：

- 每日记录提交流程
- 每日结算生成流程
- 周报生成流程

#### Domain Service Layer

职责：

- 承载核心业务规则
- 保持规则可测试、可解释
- 避免依赖 Web 层对象

典型服务：

- `ScoringService`
- `AttributeService`
- `GamificationService`
- `RouteProgressService`
- `LLMService`

#### Repository & Integration Layer

职责：

- MySQL 数据访问
- MongoDB 文档读写
- Redis 缓存和限流
- 外部 LLM API 调用

## 5. 后端模块设计

### 5.1 auth

职责：

- 用户注册
- 用户登录
- 密码加密
- JWT 签发和刷新
- 当前用户身份解析

### 5.2 user

职责：

- 用户基础信息维护
- 用户头像、邮箱等信息管理
- 用户账号状态管理

### 5.3 profile

职责：

- 目标类型设置
- 当前目标设置
- 反馈风格设置
- 成长路线绑定
- 首次进入系统时初始化用户角色属性

### 5.4 dailylog

职责：

- 接收结构化每日记录
- 接收自然语言日志
- 保存原始记录和解析后的结构化字段
- 为评分、属性变化和 LLM 反馈提供输入

### 5.5 scoring

职责：

- 计算学习成长得分
- 计算任务执行得分
- 计算精力恢复得分
- 计算情绪状态得分
- 计算娱乐控制得分
- 计算复盘质量得分
- 根据权重计算每日总分
- 根据总分生成评级
- 输出每个分数的解释依据

评分模块必须是确定性规则模块，不能依赖 LLM 生成最终分数。

### 5.6 attribute

职责：

- 维护用户角色属性
- 根据每日记录和评分结果计算属性变化
- 生成属性变化原因
- 更新等级和经验值

核心属性：

- focus
- discipline
- knowledge
- energy
- mood
- execution
- balance

### 5.7 gamification

职责：

- 根据行为模式生成游戏化事件
- 识别常见问题事件
- 生成 Buff / Debuff
- 生成每日结算事件列表
- 为 LLM 旁白提供结构化上下文

### 5.8 route

职责：

- 维护默认成长路线
- 维护章节和关卡
- 计算路线进度
- 根据每日记录推动当前路线进度

### 5.9 llm

职责：

- 封装统一 `LLMService`
- 解析自然语言日志
- 生成每日反馈
- 生成明日任务
- 生成周报
- 生成剧情化旁白
- 处理 LLM 调用失败时的降级结果

LLM 模块不得直接覆盖规则评分结果。

### 5.10 trend

职责：

- 聚合每日评分趋势
- 聚合属性变化趋势
- 聚合学习、睡眠、情绪、娱乐控制等趋势
- 为前端图表提供结构化数据

### 5.11 weekly

职责：

- 汇总一周记录
- 统计最高分、最低分、主要问题和成长点
- 调用 LLM 生成周报建议
- 在 LLM 不可用时生成规则模板周报

## 6. 核心业务流程

### 6.1 首次使用流程

```text
用户注册 / 登录
-> 选择目标类型
-> 设置当前目标、周期、计划投入时间
-> 选择反馈风格
-> 选择或绑定成长路线
-> 初始化角色属性
-> 进入成长首页
```

### 6.2 每日记录与结算流程

```text
提交每日记录
-> 保存原始日志
-> 如果存在自然语言日志，调用 LLM 解析结构化字段
-> 合并表单字段与 LLM 解析结果
-> 规则评分
-> 计算属性变化
-> 生成游戏化事件
-> 生成基础结算结果
-> 异步调用 LLM 生成反馈、旁白和明日任务
-> 前端展示结算页
-> LLM 结果完成后更新结算内容
```

### 6.3 LLM 降级流程

```text
调用 LLM
-> 成功：保存 prompt 和 response，更新反馈内容
-> 失败：记录失败原因，返回规则模板反馈
```

降级时必须保留：

- 每日评分
- 评级
- 属性变化
- 属性变化原因
- 基础建议
- 明日任务模板

### 6.4 周报流程

```text
定时或用户手动触发周报
-> 汇总最近 7 天 daily_log / daily_score / attribute_change / game_event
-> 计算趋势和高频事件
-> 生成规则版周报结构
-> 调用 LLM 生成个性化表达
-> 保存周报内容
-> 前端展示周报页
```

## 7. 数据存储架构

### 7.1 MySQL

存储结构化业务数据：

- `user`
- `user_profile`
- `daily_log`
- `daily_score`
- `user_attribute`
- `attribute_change`
- `game_event`
- `llm_feedback` 元数据

### 7.2 MongoDB

存储长文本和半结构化内容：

- 用户长文本日志
- LLM 原始 prompt
- LLM 原始 response
- 周报正文
- 长期复盘内容

MVP 阶段可以先只使用 MySQL 存储必要字段，但文档和模块设计应保留 MongoDB 扩展位。

### 7.3 Redis

用于：

- 用户近期状态缓存
- 首页成长面板缓存
- LLM 调用限流
- 登录会话辅助信息
- Token 黑名单或短期验证码

Redis 不能作为核心业务数据的唯一存储。

## 8. 接口设计原则

接口按资源和业务流程划分：

- `/api/auth/**`
- `/api/users/**`
- `/api/profile/**`
- `/api/daily-logs/**`
- `/api/settlements/**`
- `/api/routes/**`
- `/api/trends/**`
- `/api/weekly-reports/**`

通用响应结构建议：

```json
{
  "code": "OK",
  "message": "success",
  "data": {}
}
```

错误响应建议：

```json
{
  "code": "VALIDATION_ERROR",
  "message": "参数校验失败",
  "data": {
    "field": "sleepHours",
    "reason": "睡眠时长不能小于 0"
  }
}
```

## 9. 安全与隐私

### 9.1 身份认证

- 登录后使用 JWT 访问受保护接口。
- 后端从 Token 中解析用户身份。
- 所有用户数据查询必须带上当前用户 ID 作为隔离条件。

### 9.2 数据隔离

- 每日记录、评分、属性、事件、反馈和周报都必须按用户隔离。
- 禁止通过前端传入的 `userId` 直接决定访问范围。
- 管理类接口后续单独设计权限模型，MVP 阶段不开放跨用户数据访问。

### 9.3 隐私保护

- 用户原始日志、情绪和生活记录不做公开展示。
- LLM prompt 中只放入完成任务所需的最小上下文。
- 记录 LLM 原始响应时，应与用户 ID 绑定并受权限保护。

### 9.4 项目边界

项目不是心理咨询系统，也不是医疗建议系统。

当用户输入涉及严重心理危机、健康风险等内容时，系统应提示寻求专业帮助，不输出诊断或治疗结论。

## 10. 异步与性能

### 10.1 同步路径

每日记录提交后，同步完成：

- 记录保存
- 规则评分
- 属性变化
- 基础游戏化事件
- 基础结算结果

这样可以保证用户快速看到稳定结果。

### 10.2 异步路径

可异步完成：

- LLM 每日反馈
- LLM 剧情旁白
- LLM 明日任务
- LLM 周报文本

MVP 初期可以使用 Spring 异步任务，后续再引入消息队列。

### 10.3 性能目标

- 普通接口响应尽量控制在 300ms 内。
- LLM 相关接口允许更长响应时间，但不阻塞基础结算展示。
- 首页成长面板可使用 Redis 缓存近期状态。

## 11. 可测试性设计

重点测试模块：

- `ScoringService`：各维度评分和总分权重。
- `AttributeService`：属性变化、经验和等级。
- `GamificationService`：事件和状态触发规则。
- `LLMService`：成功、失败、超时和降级路径。
- 权限隔离：不同用户不能读取彼此数据。

测试原则：

- 核心规则使用单元测试覆盖。
- 业务编排使用服务层测试覆盖。
- API 层验证参数校验和鉴权。

## 12. MVP 到后续演进

### 12.1 MVP 阶段

优先实现：

- 注册登录
- 目标设置
- 每日记录
- 规则评分
- 属性变化
- 每日结算
- LLM 反馈
- 明日任务
- 趋势可视化

### 12.2 后续阶段

可扩展：

- 更复杂的成长路线分支
- 插件化路线模板
- 可穿戴设备接入
- 多 Agent 成长教练
- 社交协作能力

这些能力不应影响 MVP 的基础模块边界。
