# LifeQuest Agent Rules

本文件用于约束后续 AI Agent 或开发者在构建 LifeQuest 项目时的基础行为。任何代码生成、重构、目录调整或功能实现，都应先阅读并遵守本文件。

## 每次构建前必须阅读

每次开始开发、修改代码或调整架构前，必须先阅读以下文件：

1. `agent.md`
2. `README.md`
3. `docs/current_process.md`
4. `docs/requirements/` 下的需求资料
5. 与本次任务相关模块的源码目录

如果需求文档、架构文档和现有代码存在冲突，优先级如下：

1. 用户本次明确指令
2. `agent.md`
3. 现有代码结构和已实现逻辑
4. `README.md`
5. `docs/current_process.md`
6. `docs/requirements/` 下的需求资料

## 不可破坏的文件和目录

以下文件和目录不得随意删除、清空、覆盖或大范围改写：

- `agent.md`
- `README.md`
- `docs/current_process.md`
- `docs/`
- `backend/src/main/java/com/lifequest/`
- `backend/src/main/resources/db/migration/`
- `frontend/src/`
- `infra/`
- `.gitignore`
- `.github/workflows/`

如果确实需要修改以上内容，必须满足：

- 修改与当前任务直接相关
- 保留已有有效信息
- 不删除用户已有改动
- 不进行无关重构
- 不改变项目核心定位

## 不可违反的项目原则

LifeQuest 的核心设计原则如下，不得在实现中破坏：

- 项目定位是基于 LLM 的游戏化自我成长平台，不是普通 Todo / 打卡工具。
- 评分结果必须以规则计算为主，不能完全交给 LLM 决定。
- LLM 主要负责自然语言解析、反馈生成、明日任务、周报和剧情化表达。
- 当 LLM 调用失败时，系统仍必须能展示基础评分、属性变化、规则建议和任务模板。
- 用户隐私数据不得公开展示，日志、情绪和生活记录必须做用户隔离。
- 项目不是心理咨询或医疗建议系统，不得输出诊断、治疗结论或替代专业帮助的建议。
- MVP 优先保证完整闭环，不优先实现社交、公会、排行榜、复杂装备等扩展功能。

## 架构约束

项目采用前后端分离架构：

- `backend/`：Spring Boot 后端服务
- `frontend/`：Vue 3 + TypeScript 前端应用
- `docs/`：需求、架构、API、数据库文档
- `infra/`：MySQL、MongoDB、Redis、Docker 等基础设施配置
- `scripts/`：开发、测试、部署脚本

后端模块边界：

- `auth`：注册、登录、JWT 和权限控制
- `user`：用户基础信息
- `profile`：目标、路线、反馈风格
- `dailylog`：每日记录和自然语言日志
- `scoring`：规则评分
- `attribute`：角色属性和属性变化
- `gamification`：敌人、事件、Buff / Debuff、结算事件
- `route`：成长路线、章节和关卡
- `llm`：统一 LLM 服务封装
- `trend`：成长趋势数据
- `weekly`：周报和周期复盘
- `common`：通用响应、异常、工具类
- `config`：项目配置

前端模块边界：

- `api`：接口请求封装
- `components`：可复用组件
- `components/charts`：趋势图、雷达图等图表组件
- `components/game`：游戏化展示组件
- `components/layout`：布局组件
- `router`：路由
- `stores`：状态管理
- `styles`：全局样式
- `types`：类型定义
- `utils`：工具函数
- `views`：页面级组件

## 开发规则

- 修改前先理解现有结构，不凭空引入新的架构风格。
- 保持改动范围与当前任务匹配，避免无关重构。
- 新增模块时应放入对应目录，不应随意创建平行目录。
- 公共逻辑应放在 `common`、`utils` 或明确的共享模块中。
- 规则评分逻辑应保持可解释，每个分数或属性变化应能追溯原因。
- 重要业务规则应写入后端服务层，不应只写在前端。
- 前端页面应围绕真实工作流构建，不做只有展示意义的空壳页面。
- 数据表、接口、业务字段命名应尽量与需求文档保持一致。
- 不使用破坏性 Git 操作，例如 `git reset --hard`、强制覆盖用户改动等。
- 不提交 `.DS_Store`、日志文件、构建产物、依赖目录和本地环境文件。

## 文档规则

新增或修改重要功能时，应同步考虑是否需要更新：

- `README.md`
- `docs/api/`
- `docs/database/`
- `docs/architecture/`
- `docs/requirements/`
- `docs/current_process.md`

每次后端发生更改时，必须检查 `docs/api/mvp_api_design.md` 是否仍与实际后端架构和接口实现相符。
如果后端接口路径、请求字段、响应字段、错误码、鉴权规则、用户身份来源、核心业务流程或模块边界发生变化，且 `docs/api/mvp_api_design.md` 与实现不一致，必须在同次修改中及时更新该文档。
如果功能涉及接口字段、数据库字段、核心流程或模块边界，必须更新对应文档或在任务说明中明确记录待更新项。
每次项目取得进展后，必须在 `docs/current_process.md` 中记录已完成工作和后续 TODO。

## 测试和验证规则

实现功能后，应根据改动范围进行验证：

- 后端核心规则需要单元测试或服务层测试。
- 前端关键页面需要至少完成本地运行验证。
- 涉及评分、属性变化、游戏事件和 LLM 降级逻辑时，应覆盖正常输入和边界输入。
- 涉及用户隐私和权限隔离时，应重点验证不能跨用户读取数据。

## 禁止事项

- 禁止为了快速实现而删除已有结构。
- 禁止把所有业务逻辑堆在单个文件中。
- 禁止让 LLM 直接决定最终评分。
- 禁止在前端硬编码后端核心评分规则作为唯一来源。
- 禁止把用户原始日志作为公开展示内容。
- 禁止在没有明确需求时引入复杂社交、排行榜、公会、装备系统。
- 禁止在没有用户确认时执行破坏性命令或覆盖用户已有改动。

## 默认工作流程

每次构建项目时，推荐遵循以下流程：

1. 阅读 `agent.md` 和 `README.md`
2. 阅读 `docs/current_process.md`
3. 阅读相关需求文档和目标模块代码
4. 确认当前任务影响范围
5. 制定最小可行改动
6. 创建或修改对应文件
7. 如果后端发生更改，检查并按需更新 `docs/api/mvp_api_design.md`
8. 运行必要验证
9. 更新 `docs/current_process.md`
10. 汇总本次变更和后续建议
